Where n8n Scores Lower: Resilience

n8n — R: 4/20
n8n disclosed 12+ CVEs in the trailing 12-month period, including two at CVSS 10.0 (the maximum possible severity). CVE-2025-68613 (CVSS 9.9) was added to the CISA Known Exploited Vulnerabilities catalog in March 2026, confirming active exploitation in the wild. Censys identified over 100,000 potentially vulnerable instances at disclosure. Five or more CVEs share the same root cause: sandbox bypass in a blocklist-based architecture. The v2.0 migration to Pyodide/WASM represents an architectural acknowledgment of this limitation.
Make.com — R: 14/20
Zero CVEs were confirmed for Make.com in any period searched (NVD, GHSA). Make.com operates as a closed-source cloud platform, which limits community vulnerability discovery but also limits the public attack surface visible to researchers. No CISA KEV entries. No supply chain incidents confirmed.

Where n8n Scores Higher: Transparency

n8n — T: 6/10
n8n publishes a SECURITY.md with a defined vulnerability reporting process and a 90-day disclosure timeline. Security advisories are published through GitHub Security Advisories. The project's open-source nature provides inherent code transparency. Despite the high CVE count, the disclosure posture is documented and functional.
Make.com — T: 5/10
Make.com publishes a security page and maintains a responsible disclosure process. As a closed-source platform, code-level transparency is not available. No public security advisories were identified during the evaluation period.

Where Both Score the Same

Three dimensions produced identical scores, revealing structural similarities between the two platforms despite their different architectures (self-hosted open source vs. cloud SaaS).

Data Conduct — Both 8/15
Both platforms document GDPR compliance. Make.com publishes a DPA and sub-processor list. n8n's self-hosted option provides inherent data sovereignty (data never leaves the user's infrastructure). Neither platform has a publicly documented AI model training opt-out policy.
Identity & Control — Both 5/10
Both platforms offer RBAC and workflow-level permissions. Neither documents a formal emergency stop mechanism for runaway automations. Make.com provides SSO on enterprise plans. n8n provides user management in the self-hosted version.
Containment — Both 4/10
This is the most significant shared weakness. n8n's pre-v2.0 blocklist sandbox produced five or more bypass CVEs. Make.com's execution isolation is not publicly documented beyond the shared cloud infrastructure. Both platforms connect to hundreds of third-party services, creating a broad integration attack surface. Containment is the weakest average dimension across the VERDICT index (4.9/10).

Architecture: Different Models, Shared Risks

n8n is an open-source, self-hostable workflow automation platform (fair-code license). Make.com is a closed-source cloud SaaS. This architectural difference creates different risk profiles rather than uniformly better or worse ones.

Self-hosted n8n gives organizations full control over data residency, network configuration, and patch timing. It also means the organization is responsible for applying security patches, configuring sandboxing, and managing access control. The 100,000+ potentially vulnerable instances identified by Censys during the CISA KEV disclosure suggest many self-hosted deployments do not keep pace with security updates.

Make.com as a cloud SaaS applies patches centrally and manages infrastructure security. Users cannot inspect the execution environment, audit the codebase, or verify containment architecture. The zero-CVE record may reflect actual security strength, or it may reflect reduced visibility into a closed system.

The CISA KEV Differential

n8n is one of two platforms in the VERDICT index listed in the CISA Known Exploited Vulnerabilities catalog (the other is Langflow). CVE-2025-68613 (CVSS 9.9) allows authenticated users to execute arbitrary code with n8n process privileges. Federal agencies subject to Binding Operational Directive 22-01 must remediate KEV-listed vulnerabilities within specified deadlines.

Make.com has no CISA KEV entries. For procurement teams evaluating both platforms, this is a binary distinction: one has confirmed active exploitation in the wild, the other does not.

What This Data Shows

The comparison reveals that the security gap between n8n and Make.com is narrow in most dimensions and concentrated in vulnerability history. Teams prioritizing auditability and data sovereignty may weigh n8n's open-source transparency and self-hosting options. Teams prioritizing a clean vulnerability record and managed security may weigh Make.com's zero-CVE history and cloud-managed infrastructure.

Both platforms share the same containment weakness (4/10) that characterizes the workflow automation category. Neither has solved the fundamental challenge of limiting blast radius when a connected third-party service or a workflow execution is compromised.

VERDICT presents the data. The weighting of each dimension is a decision for the procurement team evaluating these platforms for their specific use case and risk tolerance.
